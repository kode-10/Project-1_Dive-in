# nodeway

A research map for anything you're curious about — instead of Instagram.

Type a topic (or hover "surprise me" for 5 random suggestions, or click it for
one fully random pick from 180+ topics) and get a full dossier:

- **AI Summary** — a sharp, non-encyclopedic overview, generated by whichever
  AI connection you've set active. Falls back to the Wikipedia extract if you
  haven't set one up, clearly badged either way. A **Wikipedia ↗** link next
  to the badge jumps straight to the source article.
- **Visuals** — real images from Wikipedia + Openverse (CC-licensed).
- **Primary Sources** — real papers from CrossRef (broad coverage across every
  discipline) and Semantic Scholar, merged, deduped, and **sorted by citation
  count** so the most-cited work surfaces first. The list scrolls instead of
  dumping everything on the page.
  - Click any paper: a bottom sheet slides up with an AI-generated read —
    what it's about, the findings you should expect, anything genuinely
    surprising — plus a button to open the full paper. Click the same paper
    again to retract the sheet.
- **Explore Next** — related topics from Wikipedia's related-pages API, click
  any one to jump into a new dossier.
- **Mind Map** — every topic you open becomes a node (teal = explored, indigo
  = suggested). Interactive, force-directed, drag/zoom/click.
- **The Path** — the 5-step framework for going from zero to competent in a
  niche, behind its own button.
- **The Catalogue** — every topic you've opened, logged locally.

No backend, no build step. Runs as static files.

## Run it

Open `index.html` directly, or serve it statically:

```bash
python3 -m http.server 8000
```

## Deploy it free (GitHub Pages)

1. Push this repo to GitHub.
2. Repo Settings → Pages → Source: `main` branch, root folder.
3. Live at `https://<your-username>.github.io/<repo-name>/`.

**Test on the deployed URL**, not a sandboxed preview or `file://` — some
sandboxed iframes (including Claude's own in-chat preview) restrict
cross-origin fetches, so panels can look broken there even when the code is
correct. On GitHub Pages it's a normal page with normal network access.

## AI Connections

Settings (gear icon, top right) is now a full connection manager, not a
single key field:

- **Add as many connections as you want**, from different providers, each
  with its own nickname.
- **Test** any connection right there — a status dot on that row turns green
  (working) or red (failed) after a real test call. The gear icon itself
  carries the same dot for whichever connection is currently **active**, so
  you always know at a glance whether AI features are live.
- **Active** connection (radio button per row) is the one used for AI Summary
  and paper insights. Switch anytime — no need to delete anything.
- Nothing leaves your browser except a direct call from your browser to the
  provider you choose, using your key.

**Supported today:**
| Provider | Cost | Notes |
|---|---|---|
| Google AI Studio (Gemini) | Free tier | Easiest first connection — aistudio.google.com/apikey |
| Anthropic (Claude) | Pay-as-you-go | console.anthropic.com |
| Any OpenAI-compatible endpoint | Varies | Groq (free tier, fast), OpenRouter (has free models), Together, local Ollama/LM Studio, etc. — just supply the base URL |

The "OpenAI-compatible" option is the scaling mechanism: any provider that
implements the standard `/chat/completions` shape works without new code —
that's most of the ecosystem outside Anthropic and Google. To add a brand new
bespoke provider (one with its own non-standard request/response shape),
add an adapter function in `js/connections.js` (`callAnthropic` and
`callGemini` are the two examples to copy) and register it in `PROVIDER_META`
and the `call()` switch.

If a connection fails to test, the two usual causes are a typo'd model name,
or the provider blocking direct browser calls (CORS) — some providers only
allow server-side use for security. Anthropic, Gemini, Groq, and OpenRouter
are all confirmed to work directly from the browser.

## How it works

- `js/topics.js` — curated shelf of topics for "surprise me" and search
  typeahead.
- `js/connections.js` — self-contained multi-provider connection manager:
  storage (`nodeway.connections` in localStorage), provider adapters, the
  test-connection flow, the gear-icon status dot, and the whole Settings
  modal UI. Exposes `window.NodewayAI` (`callActive(prompt)`, `getActiveConnection()`,
  `testConnection(id)`, etc.) for the rest of the app to use.
- `js/app.js` — Wikipedia primer/images/related-topics fetches, CrossRef +
  Semantic Scholar paper search (merged, deduped, sorted by citations), the
  paper insight bottom sheet (calls `window.NodewayAI.callActive` per click,
  with an in-flight guard so clicking a second paper before the first
  response lands doesn't show the wrong result), search suggestions, the
  surprise-me popover, and the catalogue. Calls into `window.NodewayAI` for
  every AI call — it never talks to a provider directly.
- `js/mindmap.js` — self-contained graph module (`nodeway.graph` in
  localStorage), rendered with D3 (`forceSimulation` + drag + zoom). Exposes
  `window.NodewayMap`; `app.js` feeds it after every search, and it calls
  back into `app.js`'s `openDossier` when you click a node.

## Roadmap (not built yet)

- Reordering connections by drag (right now priority = active selection, not
  an automatic fallback chain — if the active connection fails at call time,
  it currently falls back straight to Wikipedia/abstract rather than trying
  your next connection; a fallback chain is a natural next step).
- Author/practitioner panel via Semantic Scholar's author endpoint.
- Export the catalogue or mind map as markdown notes.
- Field-based node coloring in the mind map.
